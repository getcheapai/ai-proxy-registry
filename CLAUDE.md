# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

`ai-proxy-registry` is an npm package that centralizes metadata for AI API proxy stations (name, homepage, invite link, underlying system, opening date, etc.) so that multiple sites can share it. It is purely data + type definitions, with no runtime logic.

## Common commands

```bash
npm run build       # Vite builds ESM/CJS/d.ts, then scripts/generate-json.mjs emits dist/stations.json
npm run typecheck   # tsc --noEmit
```

Note: `stations.json` is generated by `scripts/generate-json.mjs`, which reads from the **build output** `dist/index.js`. Therefore `vite build` must run before the script (`npm run build` already wires up the order). Editing `src/` and running the script alone will not update the JSON.

## Adding a new station

1. Append an object to the **end** of the array in `src/stations.ts`, keeping the trailing `] as const satisfies readonly Station[]` intact (breaking it degrades `StationId` back to `string`).
2. Required fields: `id`, `name`, `homepage`, `statusUrl`, `privacyUrl`, `refundUrl`, `updatedAt`. The three url fields are typed `string | false` — when there is no corresponding page you must explicitly write `false`; omitting it is a compile error. The remaining fields (`system`, `openedAt`, `i18nNames`, `supportsInvoice`, etc.) are optional.
3. Run `npm run build`; `dist/stations.json` is regenerated automatically — do not hand-edit the JSON.
4. Run `npm run typecheck` to confirm types pass (missing required fields or a wrong id will fail here).

## Architecture notes

### Type-safe station id union

`src/stations.ts` uses the `as const satisfies readonly Station[]` double assertion:

- `as const` preserves the literal type of every object field, so `(typeof stations)[number]['id']` infers a union like `'yunwu' | ...`; external consumers get compile-time id checking via `StationId`.
- `satisfies readonly Station[]` guarantees every entry still conforms to the `Station` interface.

Keep this pattern when adding a station, otherwise `StationId` degrades to `string` and breaks the core selling point of the public API.

### Build pipeline

- `vite.config.ts` uses library mode (`build.lib`) to output both ESM and CJS.
- `vite-plugin-dts` with `rollupTypes: true` bundles all `.d.ts` into a single `dist/index.d.ts`.
- `scripts/generate-json.mjs` imports `stations` from the bundled `dist/index.js`, then writes `dist/stations.json`, keeping the JSON always consistent with the JS output (avoiding a separately maintained copy).

### TS configuration constraints

- `verbatimModuleSyntax: true` requires `import type` to explicitly mark type-only imports; follow this pattern in new source files.
- `moduleResolution: "Bundler"`, but internal relative imports still carry the `.js` suffix (e.g. `'./stations.js'`) to stay compatible with the compiled output's ESM resolution.
- `noUncheckedIndexedAccess: true`: indexed access such as `i18nNames['zh-CN']` returns `T | undefined`, so handle the null case.

### Package export convention

`package.json`'s `exports` exposes both the main entry and the `./stations.json` subpath, and the `files` field only publishes `dist`. When changing the export structure, update both places, otherwise consumers cannot access the new files.
